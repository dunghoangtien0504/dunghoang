## 🔴 CRITICAL

### [C1] HARDCODED-SECRET — Mật khẩu admin bị hardcode trong source code

**Files vi phạm:**
- `app/api/admin/login/route.ts:9-10`
- `middleware.ts:31`

**Code lỗi:**
```ts
// app/api/admin/login/route.ts
const expectedPassword = process.env.ADMIN_PASSWORD || 'Dung687568!'

// middleware.ts
const secret = process.env.ADMIN_PASSWORD || 'Dung687568!'
```

**Tác động:**
- Mật khẩu tồn tại vĩnh viễn trong git history — bất kỳ ai clone repo đều thấy
- Nếu `ADMIN_PASSWORD` env var không được set trong bất kỳ môi trường nào (staging, preview deploy), password cứng tự động được dùng
- Token signing cũng dùng ADMIN_PASSWORD làm secret key → nếu password lộ, attacker có thể forge admin tokens mà không cần brute force

**Cách sửa — 2 bước bắt buộc:**

Bước 1: Đổi password ngay trong Vercel Dashboard (Settings > Environment Variables) — đổi sang giá trị mới, mạnh hơn.

Bước 2: Xóa fallback hardcode trong cả 2 file — throw error nếu thiếu env var:

```ts
// app/api/admin/login/route.ts
const expectedUsername = process.env.ADMIN_USERNAME
const expectedPassword = process.env.ADMIN_PASSWORD
if (!expectedUsername || !expectedPassword) {
  return NextResponse.json({ error: 'Server misconfigured' }, { status: 500 })
}

// middleware.ts
const secret = process.env.ADMIN_PASSWORD
if (!secret) {
  return NextResponse.json({ error: 'Unauthorized' }, { status: 401 })
}
```

Bước 3 (optional, nâng cao): Tách `TOKEN_SECRET` riêng khỏi `ADMIN_PASSWORD` để 2 secrets độc lập với nhau.

## 🟠 HIGH

### [H1] MISSING-RATE-LIMIT — Admin login không có giới hạn thử

**File:** `app/api/admin/login/route.ts`

**Mô tả:**
Endpoint `/api/admin/login` không có rate limiting. Attacker có thể thử mật khẩu không giới hạn số lần. Risk tăng gấp đôi vì C1 — password fallback đã bị expose trong source code, attacker biết điểm khởi đầu để thử.

**Cách sửa — tùy level:**

Option A (đơn giản nhất, không cần thêm package):
```ts
// Thêm delay cố định sau mỗi lần thất bại
await new Promise(r => setTimeout(r, 2000))
return NextResponse.json({ error: 'Sai tên đăng nhập hoặc mật khẩu' }, { status: 401 })
```

Option B (production-grade, dùng Upstash Redis):
```ts
import { Ratelimit } from '@upstash/ratelimit'
// Max 5 lần thất bại / 15 phút per IP
```

### [H2] XSS — dangerouslySetInnerHTML với content từ database

**Files vi phạm:**
- `app/(portal)/portal/[slug]/page.tsx:193`
- `app/(portal)/portal/[slug]/[lesson]/page.tsx` (pattern tương tự)

**Code lỗi:**
```tsx
<div
  className="prose prose-sm max-w-none"
  dangerouslySetInnerHTML={{ __html: lesson.content_html }}
/>
```

**Phân loại data flow:** L2 (Supabase DB) — hiện tại chỉ admin insert qua SQL, không phải user input.

**Tác động hiện tại:** Thấp — content hoàn toàn do bạn kiểm soát.

**Tác động nếu DB bị compromise (kết hợp C1):** Cao — attacker có thể inject script ảnh hưởng toàn bộ học viên đang đăng nhập, đánh cắp session cookie, Supabase auth token.

**Cách sửa:**
```bash
npm install isomorphic-dompurify
npm install @types/dompurify
```
```tsx
import DOMPurify from 'isomorphic-dompurify'

<div
  className="prose prose-sm max-w-none"
  dangerouslySetInnerHTML={{ __html: DOMPurify.sanitize(lesson.content_html ?? '') }}
/>
```

## 🟡 MEDIUM

### [M1] MISSING-SECURITY-HEADERS — Public routes thiếu security headers

**Files liên quan:** `middleware.ts`, `next.config.ts`

**Mô tả:**
Security headers hiện chỉ được thêm vào response cuối cùng của middleware — nhưng code logic trả về `NextResponse.redirect()` sớm cho unauthenticated requests, KHÔNG đi qua đoạn set headers. Kết quả: các public routes như `/`, `/24-ai-agent`, `/coaching`, `/portal/login` đều không có headers bảo mật.

Ngoài ra, không có Content Security Policy (CSP) ở bất cứ đâu trong toàn bộ app.

**Cách sửa — di chuyển headers lên đầu middleware:**
```ts
export async function middleware(request: NextRequest) {
  // Set security headers TRƯỚC MỌI logic
  const response = NextResponse.next()
  response.headers.set('X-Frame-Options', 'DENY')
  response.headers.set('X-Content-Type-Options', 'nosniff')
  response.headers.set('Referrer-Policy', 'strict-origin-when-cross-origin')
  // ... rồi mới check auth, redirect, v.v.
}
```

Hoặc tốt hơn, đặt vào `next.config.ts`:
```ts
async headers() {
  return [{
    source: '/(.*)',
    headers: [
      { key: 'X-Frame-Options', value: 'DENY' },
      { key: 'X-Content-Type-Options', value: 'nosniff' },
      { key: 'Referrer-Policy', value: 'strict-origin-when-cross-origin' },
    ],
  }]
}
```

**Cách sửa:** Thêm IP-based rate limit. Với Vercel, cách đơn giản nhất là dùng Vercel Edge Config hoặc thêm package `@upstash/ratelimit` + Upstash Redis free tier.

### [M3] UNVALIDATED-PAGINATION — listUsers() thiếu perPage trong webhook thanh toán

**File:** `app/api/webhooks/sepay/route.ts:111`

**Code lỗi:**
```ts
const { data: existingUsers } = await supabaseAdmin.auth.admin.listUsers()
// Các chỗ khác trong codebase đều dùng { perPage: 1000 }
```

**Tác động:** Supabase mặc định trả về page đầu tiên (thường 50-100 users). Nếu database có nhiều user, webhook có thể không tìm thấy email đã tồn tại → tạo duplicate account → activate order 2 lần cho cùng 1 người.

**Cách sửa (1 dòng):**
```ts
const { data: existingUsers } = await supabaseAdmin.auth.admin.listUsers({ perPage: 1000 })
```

## 🟢 LOW

### [L1] MISSING-ENV-EXAMPLE — Không có file .env.example

Không có `.env.example` trong repo. Developer mới setup sẽ không biết cần set những env vars nào → dễ bỏ sót ADMIN_PASSWORD → fallback hardcode (C1) tự động được dùng.

**Cách sửa:** Tạo `.env.example` với placeholder values (KHÔNG đặt giá trị thật):
```
NEXT_PUBLIC_SUPABASE_URL=https://your-project.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=your-anon-key
SUPABASE_SERVICE_ROLE_KEY=your-service-role-key
ADMIN_USERNAME=your-admin-username
ADMIN_PASSWORD=your-strong-password
SEPAY_WEBHOOK_SECRET=your-webhook-secret
CRON_SECRET=your-cron-secret
RESEND_API_KEY=your-resend-api-key
NEXT_PUBLIC_SITE_URL=https://dunghoang.com
NEXT_PUBLIC_META_PIXEL_ID=your-pixel-id
```

Đảm bảo `.env.local` và `.env` đang được gitignore (kiểm tra `.gitignore`).

### [L2] ADMIN-LAYOUT-CLIENT — Admin layout không có server-side auth guard

**File:** `app/admin/layout.tsx`

Layout là `'use client'` component, không có server-side auth check. Auth được xử lý hoàn toàn bởi `middleware.ts` — đây là pattern hợp lệ với Next.js App Router. Hiện tại middleware protect đúng các routes `/admin` và `/api/admin`.

**Rủi ro:** Nếu middleware config bị thay đổi sai (ví dụ ai đó xóa `/admin` khỏi `PROTECTED_PREFIXES`), không có lớp bảo vệ thứ hai ở server.

**Cách sửa (optional):** Thêm một server component wrapper kiểm tra cookie:
```ts
// app/admin/AuthGuard.tsx (server component)
import { cookies } from 'next/headers'
import { redirect } from 'next/navigation'
import { verifyToken } from '@/lib/auth-token'
```
